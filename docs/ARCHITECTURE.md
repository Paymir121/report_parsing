# Архитектура проекта report_parsing (исправленная)

Десктопное приложение для заполнения Word-шаблонов (Jinja2/docxtpl) и генерации документов с **управлением данными**: данные для подстановки в шаблоны могут поступать из **импорта Excel** (в БД), из **ручного ввода/редактирования** в UI, либо вводиться напрямую в таблице полей при генерации. Настройки интерфейса и БД, геометрия и состояние главного окна хранятся в QSettings.

---

## Часть 1. Исправленная архитектура

### 1.1. Краткое понимание задачи

- Приложение не только генерирует документы из Word-шаблонов, но и **управляет данными**, подставляемыми в шаблоны.
- Значения для `{{ FIELD }}` могут браться из:
  1. **Ручного ввода** в таблице полей при генерации (как сейчас).
  2. **Сохранённых в БД наборов данных** (таблицы/справочники): клиенты, договоры, сотрудники и т.д.
  3. **Импорта из Excel**: загрузка .xlsx → данные попадают в БД (в виде записей) → затем используются при генерации (выбор записи или подстановка).
- Должны быть поддержаны оба сценария: **импорт из Excel** и **ручное создание/редактирование записей** в этих таблицах без Excel.

### 1.2. Что не хватало в исходном плане

- Модель данных описывала только **справочник значений по полям** (`field_values`): плоский список вариантов для каждого поля шаблона.
- Не было сущностей для **наборов записей** (таблиц данных): одна запись = одна строка с множеством полей (клиент, договор, акт и т.д.).
- Импорт из Excel рассматривался как «подстановка имени/значения в таблицу полей», а не как **загрузка записей в БД** с возможностью выбора записи при генерации.
- Не было разделения: **метаданные шаблонов** | **справочные/вводимые данные (записи)** | **история генерации**.

### 1.3. Исправленная архитектура приложения

- **Слой БД**: метаданные шаблонов, поля шаблонов, справочники значений полей (`field_values`), **наборы данных (таблицы)**, **записи** и **значения записей**, история генерации.
- **Сервисы**: работа с шаблонами, с наборами данных и записями, **импорт Excel в БД**, генерация документа (контекст по-прежнему собирается в UI и передаётся словарём).
- **UI**: главное окно (шаблон → поля/значения → генерация), возможность **выбора записи из БД** для подстановки в поля, **диалоги**: управление шаблонами, управление наборами данных, импорт Excel, ручное редактирование записей.

### 1.4. Исправленная архитектура БД: сущности и связи

| Таблица | Назначение |
|--------|------------|
| **document_templates** | Шаблоны: имя, code, описание, путь/содержимое, флаг хранения в БД. Опционально: **linked_data_table_id** — привязка к набору данных для выбора записей. |
| **template_versions** | Версии шаблона (при необходимости). |
| **template_fields** | Поля шаблона: field_name, display_name, тип, обязательность, sort_order. |
| **field_values** | Справочные значения для полей (для выбора в UI по одному полю). |
| **data_tables** | Наборы данных (справочники): имя, code, описание. Например: «Клиенты», «Договоры». |
| **data_table_columns** | Колонки набора данных: field_name, display_name, sort_order. Определяют структуру записей и маппинг с Excel. |
| **data_records** | Запись в наборе данных (одна строка: один клиент, один договор и т.д.). |
| **data_record_values** | Значения полей записи: data_record_id, field_name, value_text. Пары (record, field_name) уникальны. |
| **generation_history** | История генераций: template_id, output_path, context_snapshot, дата, created_by. |

Связи:

- Шаблон → поля (cascade delete); поле → field_values (cascade delete).
- Шаблон → опционально одна **data_table** (linked_data_table_id, SET NULL при удалении таблицы).
- data_table → data_table_columns (cascade delete); data_table → data_records (cascade delete).
- data_record → data_record_values (cascade delete).

Для SQLite используется тип `JSON` вместо `JSONB`. Таблица **imports** (история импорта Excel) опциональна для MVP; при необходимости: data_table_id, file_name, imported_at, rows_count, errors (JSON).

### 1.5. Excel-импорт и ручной ввод

- **Импорт Excel**: пользователь выбирает (или создаёт) набор данных → выбирает .xlsx → первая строка = заголовки (маппинг на field_name); последующие строки = записи. Для каждой строки создаётся **data_record** и набор **data_record_values** (field_name → value). Колонки Excel сопоставляются с **data_table_columns** (по имени заголовка или по созданию новых колонок при первом импорте). Пустые строки пропускаются; дубликаты записей не обязаны обрабатываться в MVP (всегда вставка новых записей).
- **Ручной ввод**: пользователь открывает список записей выбранного набора данных → создаёт новую запись или редактирует существующую → вводит/меняет значения по колонкам (data_table_columns). Сохранение → создание/обновление **data_record** и **data_record_values**.

Импорт и ручной ввод работают с **одной и той же моделью**: data_tables + data_records + data_record_values.

### 1.6. Связь данных с шаблонами Word

- У шаблона может быть указан **linked_data_table_id**. В UI при выборе такого шаблона можно показать список записей этой таблицы; при выборе записи контекст для полей шаблона собирается из **data_record_values** (по field_name). Имена полей шаблона и имена колонок/полей записей должны совпадать (или маппинг задаётся в UI — в MVP достаточно совпадения имён).
- **field_values** по-прежнему используются для подсказок/выпадающих списков по одному полю. Записи из data_tables подставляют сразу много полей.
- При генерации итоговый контекст — всегда словарь `field_name → value`: либо собран из выбранной записи, либо из ручного ввода в таблице полей, либо комбинация (запись заполнила, пользователь подправил).

### 1.7. Рекомендуемый вариант модели данных

Рекомендуется **вариант C (гибрид)**:

- **Ручной ввод** возможен напрямую в таблице полей при генерации и в экране редактирования записей (создание/редактирование записей в наборе данных).
- **Excel** импортирует данные в те же сущности (data_tables → data_records → data_record_values).
- **При генерации** пользователь может: выбрать запись из БД (подстановка в таблицу полей), либо ввести значения вручную, либо скомбинировать (выбрать запись и подправить).

Так достигается единая модель данных, понятная пользователю и удобная для desktop-приложения: один экран «таблиц данных», один способ хранения записей, два способа заполнения (Excel и ручной ввод) и один способ потребления (выбор записи + ручная правка при генерации).

## 4. Поток данных (обновлённый)

1. **Старт**: создание таблиц, загрузка списка шаблонов.
2. **Выбор шаблона**: загрузка полей, заполнение таблицы (Поле / Значение). Если у шаблона есть linked_data_table — показать выбор записи из этого набора; при выборе записи подставить в таблицу полей значения из data_record_values.
3. **Источники значений**: (а) ручное редактирование в таблице полей; (б) «Вставить из Excel» — по-прежнему может подставлять в таблицу из файла (legacy); (в) **«Импорт в БД»** — загрузка .xlsx в выбранный набор данных (data_tables); (г) **«Выбрать запись»** — подстановка из выбранной записи БД.
4. **Генерация**: контекст собирается из таблицы полей → `generate_document(template_id, context, output_path)` → рендер docxtpl, запись в generation_history.

## 5. Зависимости ключевых модулей

- **main** — без изменений.
- **main_window** — + services.data_table_service, services.excel_import_service; при выборе шаблона запрашивать linked_data_table и записи; кнопки/диалоги: управление наборами данных, импорт Excel, выбор записи.
- **services** — data_table_service использует db.models (DataTable, DataRecord, …); excel_import_service использует data_table_service и openpyxl; template_service — при необходимости отдаёт linked_data_table_id; generation_service без изменений (принимает context dict).
